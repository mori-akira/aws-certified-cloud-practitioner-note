第2章 クラウドのコンセプト

2-1 AWSクラウドの利点

セキュリティ

様々な基準への対応、第三者機関による監査、データの暗号化など。

信頼性

ハードウェアやソフトウェアの故障や不具合により、データの消失やシステムの故障が発生しにくい状態。 また、故障した状態から早期に復旧できる能力。

可用性

壊れにくさ、稼働している状態を維持する能力。
AWS Lambda ⇒ マネージドサービス
Amazon EC2 ⇒ アンマネージドサービス

伸縮性

必要に応じてリソースを増減できること。 英語でelasticという。

俊敏性

必要なリソースをすぐに準備できること。 また、学習済みもモデルを利用したAIサービスやビッグデータ分析などの既存サービスを利用し、準備時間を短縮できること。

従量課金モデル

利用した時間に応じて課金されるモデルのこと。

スケーラビリティ

拡張可能なこと。

規模の経済性

利用者が日々増え続けることで効率化が進み、結果として継続的な値下げが行われていること。

2-2 クラウドの経済性

TCOについて

オンプレミスとクラウドのコストを比較する際、コスト構造が異なるため、単純比較ができない。 その際に用いるべき考え方としてTCO(Total Cost of Ownership:総所有コスト)がある。 これはある程度の期間を想定した初期コストとランニングコストを加味したコストである。

AWSならではのコスト削減の考え方

  • オンプレミスからAmazon EC2への移行
  • マネージドデータベースサービスの利用
  • サーバレスコンピューティングの利用
  • SaaSサービスの利用

2-3 クラウドアーキテクチャの設計原則

Design for failure

Amazon.comのCTOであるWerner Vogles氏の「Everything fails, all the time」という言葉をベースとした設計原則。

Well-Architectedフレームワーク

AWSを活用してシステム設計を行う上で、指標となるフレームワーク。 次の6つの柱と呼ばれるジャンルごとの設計原則とベストプラクティスで構成される。

  • オペレーションエクセレンスの柱
  • セキュリティの柱
  • 信頼性の柱
  • パフォーマンス効率の柱
  • コスト最適化の柱
  • 持続可能性の柱

3-4 モニタリング

Amazon CloudWatch

AWS上に構築した仮想マシンやデータベースなどのリソースや、オンプレミス環境のサーバーなどの状態を監視するサービス。 特に設定しなくても情報収集が開始する。
収集する標準データを標準メトリクスと呼ぶ。 また、API経由でカスタムして書き込む情報をカスタムメトリクスと呼ぶ。
このサービスは、以下の機能を有する。

  • メトリクスの収集・監視
  • ログの収集・検索
  • アラームとアクション
  • ダッシュボード

AWS Config

リソースの設定変更をモニタリングしたり、リソースに対するセキュリティやコンプライアンスを評価するサービス。
評価ルールを設定することで、設定に対するモニタリングや通知を設定できる。 また、修復アクションを設定することで、リソースの自動修復を行う。 さらに、リソースの設定変更をタイムラインで確認することができる。

AWS CloudTrail

AWSに対する操作(API実行)を記録するサービス。

AWS License Manager

保有するソフトウェア・ライセンスとその利用状況を管理するサービス。

3-5 アクセス管理

IAM

以下を管理するサービス。

  • IAMユーザ
  • IAMグループ
  • IAMポリシー
  • IAMロール

AWSユーザについて

AWSのユーザには、大きく以下の種類が存在する。

  • ルートユーザ
    AWSアカウントを作成すると利用できるユーザ。 登録時のメールアドレスとパスワードでサインインする。 以下のルートユーザでしか行えない操作を行える。
    • AWSアカウントの支払方法を変更する
    • AWSアカウントの名前を変更する
    • サポートプランの変更
    • AWSアカウントの削除
  • IAMユーザ
    AWSアカウント内に作成できる、認証情報を持つリソース。 最初はアクセス権限を何も持たない。

IAMグループ

IAMユーザの集合。 アクセス権限をまとめて切り替える場合などに用いる。
IAMグループは他のIAMグループに所属できない。

IAMポリシー

IAMユーザやIAMグループ、IAMロール、AWSのサービスに対して、付与するアクセスの許可や拒否を設定したドキュメント。 JSONフォーマットで記述を行う。

IAMロール

IAMユーザやAWSサービスが一時的に別のアクセス権を利用できるようにする仕組み。
主に、複数のIAMポリシーをまとめる際に使用する(※独自理解)。

AWS Systems Managerのパラメータストア

Key-Value形式でデータを保存することができ、従来の環境変数の代替手段としてパラメータをクラウド上で管理するサービス。

AWS Secrets Mangaer

認証情報などの秘密情報を保管し、許可されたユーザのみがアクセスできるようにしたサービス。

3-6 仮想ファイアウォール

セキュリティグループ

AWSリソースをグループ化し、グループに対するインバウンド/アウトバウンド通信を制御するファイアウォール。
ステートフルファイアウォールとして機能する。

ネットワークACL

Amazon Cognito

アプリケーションにユーザ認証機能やアクセス管理機能を追加するサービス。
アプリケーションにユーザ認証機能を追加するユーザプールや、 アプリケーションやその利用者にAWSへの一時的なアクセス権限を提供するIDプールの機能などがある。

AWS WAF

WAFを提供するサービス。以下のサービスで利用可能。

3-8 セキュリティを評価する

Amazon EC2の料金体系

料金体系 説明
オンデマンドインスタンス EC2インスタンスが起動している時間に対して課金する体系。
スポットインスタンス AWS上で余っているEC2のキャパシティを、オンデマンドと比較して最大90%安く利用できる体系。
AWS側のキャパシティが不足してきた場合などに、利用中のインスタンスが中断される可能性がある。
リザーブドインスタンス 特定のEC2インスタンスに対する課金ではなく、1年または3年の利用を確約することで、 最大72%の割引を受けることのできる体系。
購入した条件にマッチするインスタンス料金を定額で使うことができるが、 マッチするインスタンスが存在しない場合も料金が発生してしまう点は注意が必要。
Savings Plans 特定のEC2インスタンスに対する課金ではなく、1年または3年の利用を確約することで、 最大72%の割引を受けることのできる体系。 1時間当たりどれくらいの利用料金を予定しているかを条件として購入する。
Compute Savings Plansは、途中でインスタンスファミリーを変更することができ、 AWS LambdaやAWS Fargateに対して利用できる。
EC2 Instance Savings Plansは、EC2専用で、同一インスタンスファミリー間で自由にインスタンスタイプを変更できる。
リザーブドインスタンスと同様に長期的な利用に向いているが、より柔軟な構成で利用が可能。
ハードウェア専有インスタンス EC2インスタンスが起動するホストを自分たちのAWSアカウントのインスタンスだけで独占し、ホストを専有的に利用する体形。 ホストの指定は不可。
Dedicated Hosts インスタンスを起動するホストを指定できる体系。 最も高い料金プラン。

5-3 Amazon EC2 Auto Scaling

Amazon EC2 Auto Scalingとは

EC2インスタンスのスケールインとスケールアウトを自動で行う仕組み。 インスタンスの状態をAmazon CloudWatchで収集し、アラームが発報されることで自動スケールを行う。

起動テンプレート

Auto Scalingによって起動するインスタンスの要件(AMI、インスタンスタイプなど)を定義するテンプレート。 通常のEC2インスタンス起動時にも選択できる。

Auto Scalingグループ

Auto Scalingによって起動されたインスタンスのグループ。
希望する容量、最小容量、最大容量を設定することができ、その容量に合わせてスケーリングを行う。

スケーリングポリシー

Auto Scalingが実行されるタイミングや条件を設定する仕組み。 次の3つのタイプがある。

ポリシー 説明
動的スケーリングポリシー 負荷の変動が予測できない場合に利用する。アラームに条件を動的に設定し、発報されるとスケーリングを実行する。
予定されたアクション 指定した日時や周期でスケーリングを実行するもので、負荷のタイミングが予め予測可能な場合に使用する。
予測スケーリングポリシー 使用するメトリクスを指定することで、過去14日分のデータから機械学習によって負荷を予測し、スケーリングを起こなう。 細かい設定が不要だが、急激な負荷上昇は予測できない。

5-4 Elastic Load Balancing

Elastic Load Balancingの種類

種類 説明
Application Load Balancer HTTPまたはHTTPSリクエストを対象とした負荷分散を行う。 WebサーバやWebアプリケーションに特化する。
Network Load Balancer TCPやUDPプロトコルをベースに負荷分散を行う。 HTTP以外のプロトコルを利用する場合や、静的なグローバルIPアドレスを必要とする場合に利用する。 1秒間に数百万リクエストを処理する。
Gateway Load Balancer ネットワーク同士の接続に対する負荷分散を行う。 侵入検知やウイルスチェックなどで、サードパーティ製のセキュリティツールを通したい場合などに使用する。
Classic Load Balancer 過去の互換性維持のために残されていて、新規では使用しない。

Amazon EBSのスナップショット

Amazon EBSは任意のタイミングでスナップショットを作成でき、バックアップを取ることができる。 これは増分スナップショットの仕組みをとる。
スナップショットはAmazon S3に保存される。 この時、スナップショットの一部を削除してもその分が他のスナップショットに移動し、 全てのスナップショットがなくなるまでスナップショットが維持される仕組みがとられている。

インスタンスストア

6-3 Amazon Simple Storage Service (Amazon S3)

Amazon S3の特徴

  • 99.999999999%(イレブンナイン)の耐久性
  • 容量が無制限
  • Webからアクセス可能

アクセスコントロール

方式 説明
ACL バケットまたはオブジェクトの単位で、パブリックまたは特定のAWSアカウントに公開するかどうかを設定できる。

ブロックパブリックアクセス

サーバーサイド暗号化に使用できる暗号鍵の方式

第7章 ネットワークサービス

7-2 Amazon Virtual Private Cloud

VPCの構築

VPCの作成はリージョンの選択から始まる。 1つのリージョン内にはデフォルトで5個までのVPCを作成することができる。
サブネット内に割り当てるIPアドレスのうち、5個はAWSによって利用される。 サブネットのIPアドレスの範囲は後から変更できない。 また、サブネットにはルートリソースと呼ばれるリソースが1つ設定され、経路情報を管理する。

インターネットゲートウェイ

VPCとインターネットを接続するための仮想ルータとなるゲートウェイ。

NATゲートウェイ

パブリックサブネット内に設置されるリソース。 プライベートサブネットのルートテーブルのデフォルトルートとして設定することで、 パブリックサブネットを経由してインターネットへのアウトバウンド通信が可能となる。

仮想プライベートゲートウェイ

オンプレ環境とハイブリッド接続を行うためのゲートウェイ。

サイト間VPN

オンプレのネットワークとVPCのネットワークを接続するインターネットVPNのこと。 オンプレ側にIPSecに対応したVPNルータを用意し、グローバルIPアドレスを設定する。 VPC側は仮想プライベートゲートウェイとの間でVPNを構築する。 一つの仮想プライベートゲートウェイで複数のオンプレ環境とVPNを構築できる。

クライアントVPN

PCから直接VPN接続を行う機能。 外出先からEC2インスタンスに接続する際などに用いる。

AWS Direct Connect

VPCとの間にプライベート回線を開設する機能。 オンプレ環境とDirect Connectロケーションとの間に専用線接続を用意する。 Direct ConnectロケーションはAWSが世界各地に設けている接続拠点である。

第8章 データベース・その他サービス

8-1 Amazon Relational Database Service

Amazon RDSの特徴

  • 商用またはオープンソースのDBMSに対応
  • 自動メンテナンス
  • マルチAZを利用したプライマリ・セカンダリ構成による高可用性
  • 読み取り専用サーバによるパフォーマンス向上(リードレプリカ)
  • 自動バックアップ
  • サーバーサイド暗号化
  • あとから性能変更可能

バックアップ

自動と手動によるスナップショット機能が提供される。
自動スナップショットは1日に1回、指定した時間に最大35日分のスナップショットを自動で作成する。 バックアップ保存期間を0に設定すると、自動バックアップは行われない。

Amazon Aurora

Amazonが独自に開発した高性能DBMS。 以下の特徴がある。

  • MySQLとPostgreSQLのI/F互換
  • MySQLの最大5倍のスループット、PostgreSQLの最大3倍のスループット
  • 低レイテンシのリードレプリカ
  • マルチリージョン対応
  • サーバーレス

8-2 Amazon DynamoDB

Amazon DynamoDBの特徴

  • リージョンベース
  • サーバーサイド暗号化が自動
  • マルチAZの分散配置が自動
  • 常にデータをバックアップ

AWS Database Migration Service

オンプレ環境やEC2インスタンスで稼働中のDBから、Amazon RDSやAmazon DynamoDBへのデータ移行をサポートするサービス。 RDBからNoSQLなどの異種データベース間の移行もサポートする。
データ移行の方法には一括移行と継続的移行の2種類がある。

AWS Schema Conversion Tool

異なるDBMSへデータを移行する際の、スキーマ変換をサポートするデスクトップアプリケーション。

第9章 請求と料金モデル

9-1 料金モデル

3つの無料枠

  • 常に無料のサービス
  • 12カ月無料のサービス
  • トライアル

各サービスの課金モデル

サービス 課金モデル
Amazon EC2 インスタンスの起動時間に対する従量課金。 リザーブドインスタンス、Compute Savings Plans、スポットインスタンス等を活用すると料金が下げられる可能性がある。
AWS Lambda 実行時間とリクエスト数にに対する課金がある。 どちらも無料利用枠がある。
Amazon S3 保存した容量、転送量、リクエスト数に対する課金がある。 また、保存した容量によってボリュームディスカウントが適用される。
Amazon EBS 確保した容量に対して課金される。 未使用でも課金されるため注意が必要。
Amazon DynamoDB データの読み書きと保存した容量に対して課金される。 25GBまでは保存料金は無料枠となる。
Amazon RDS 裏で動いているEC2インスタンスの起動時間と確保した容量に対して課金される。 OracleやSQL Serverを利用する場合、ライセンス料も料金に含まれる。

9-2 マルチアカウントの請求

AWS Organizations

AWSアカウントを組織としてまとめる仕組み。
AWSアカウントをOrganizations Unit(OU)というグループに所属させて管理する。 また、所属するAWSアカウントにはサービスコントロールポリシー(SCP)というアクセス権限の許可リストを設定し、権限の管理を行う。 AWS Organizationsで管理するAWSアカウントには、管理アカウントとリソースアカウントがある。
AWS Organizationsのアカウントに対する請求をまとめる機能を、一括請求という。
AWS Organizationsでは、EC2のリザーブドインスタンスで余っているインスタンスや、 S3のボリュームディスカウントを共有することができる。
